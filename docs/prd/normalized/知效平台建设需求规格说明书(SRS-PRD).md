### 1.3 产品摘要

知效平台用于替代现有知微系统，承载研发需求价值流、任务协作、外包报工、阶段时效和月度基线管理，并与知脉、知测、知安、知悉、考勤和数据仓库协同。产品目标不是复制全部旧系统界面，而是在保持考核和业务连续性的前提下，减少人工状态维护并提升过程可视化能力。

### 1.4 本期产品范围

- 需求管理：泳道、列表、甘特图、阶段流转、筛选、横幅、风险和自定义字段；
- 任务管理：需求类、项目类、事务类和安全类任务，以及分发、认领、状态和点亮；
- 报工管理：考勤同步、默认工时、个人报工、管理查看和知脉写回；
- 时效与基线：阶段停留、审批扣减、基线、取消、加塞、调出和计划分析；
- 运营保障：数据迁移、双系统灰度、回滚、安全、审计、性能、容灾和可观测性。

## 3. User Scenarios

### SCN-AUTH-001：用户登录并获得正确工作视角

**Actors**：所有交互用户、知悉统一认证平台。

**用户目标**：用户通过行内统一身份进入知效，只看到其角色和组织范围内的菜单、指标与数据。

**触发条件**：用户访问知效平台。

**前置条件**：用户在统一身份平台有效；组织、角色和人员映射已同步。

**主流程**：

1. 知效将未认证用户引导至知悉认证。
2. 知悉完成认证并回调知效。
3. 知效验证回调和令牌，加载用户角色、组织和业务关联。
4. 系统建立会话并展示用户有权访问的工作视角。
5. 用户切换页面、组织或对象时，服务端再次按作用域授权。

**Error Scenarios**：令牌无效或过期时拒绝登录并引导重新认证；用户未完成人员映射时不授予业务数据访问权，并产生可处理异常；尝试访问越权对象时返回无权限而不是空泛系统错误。

**Edge Cases**：用户拥有多个角色或跨组织兼职时，按批准的角色合并和最小权限规则计算；该优先级由`TBD-SEC-001`关闭。

**关联需求**：NFR-SEC-001～002、IF-082～092。

**成功结果**：合法用户进入正确视角，越权用户无法通过页面或直接接口获得数据。

### SCN-REQ-001：管理和查看需求价值流

**Actors**：管理层、PMO、项目经理、团队长/小队长、开发和测试人员。

**用户目标**：从同一需求集合中按泳道、列表或甘特图理解当前阶段、计划、风险和超期情况。

**触发条件**：用户进入需求管理或从横幅指标钻取。

**前置条件**：用户已登录；知脉需求已同步；数据权限可计算。

**主流程**：

1. 系统按用户权限加载在建需求和当年上线需求。
2. 用户在泳道、列表和甘特图之间切换，当前权限和筛选条件保持一致。
3. 用户按文本、枚举、日期、数值以及AND/OR组合条件筛选需求，并可保存个人筛选。
4. 用户配置列表显示列、顺序和列宽，配置仅对本人有效。
5. 系统展示需求总数、超期/受阻数、在建紧急数和组织大盘。
6. 用户点击横幅指标，列表展示对应明细；管理者可在获授权组织范围内切换视角。
7. 用户进入详情查看阶段历史、任务、风险、备忘录、附件和操作记录。

**Error Scenarios**：筛选字段或操作符非法时拒绝查询并指出问题条件；无权切换的组织不出现在选项中，直接请求也被拒绝；来源需求重复到达时更新已有映射而不创建重复需求。

**Edge Cases**：归档需求默认不进入在建视图；需求缺少计划时间时可出现在列表/泳道，但甘特图应明确标记数据不完整；自定义字段归档后保留历史值但不允许新增有效值。

**关联需求**：FR-REQ-001～005、FR-REQ-007～008、FR-REQ-011～017。

**成功结果**：同一权限和筛选下三种视图的数据集合一致，横幅统计可追溯到明细。

### SCN-REQ-002：需求阶段自动推进、人工回退与风险处理

**Actors**：知测、知脉、PMO、项目经理、部落长、风险责任人。

**用户目标**：减少人工拖卡，同时保证阶段变化、回退原因和风险处置完整留痕。

**触发条件**：收到测试/上线事件，或授权用户发起人工阶段调整。

**前置条件**：需求存在且有唯一当前阶段；外部事件具有业务标识和版本。

**主流程**：

1. 知测SIT准入通过后，系统将需求从“待SIT”推进至“SIT中”。
2. 知测UAT准出通过后，系统将需求从“UAT中”推进至“待上线”。
3. 知脉通知上线完成后，系统将需求推进至“已上线”。
4. 每次流转关闭当前阶段区间、创建目标阶段区间并写入操作日志。
5. 授权用户需要回退时，只能回退到相邻前序阶段，并填写原因。
6. 用户创建风险问题，指定类型、严重度和责任人；责任人解决后由授权人员关闭。
7. 上线满90个自然日后，J13自动归档需求。

**Error Scenarios**：重复事件不产生重复阶段记录；乱序或与当前阶段不兼容的事件进入同步异常；无权或无原因的回退被拒绝；外部对象不存在时不创建孤立阶段数据。

**Edge Cases**：同一阶段可能因回退被多次进入，每个时间区间都必须保留；除三条已确认自动拖卡外，其余阶段事件由`TBD-REQ-001`确认；归档后需求默认只读。

**关联需求**：FR-REQ-006、FR-REQ-009～011、FR-REQ-017、FR-CROSS-001～002。

**成功结果**：需求阶段历史可以还原，重复/异常事件不会破坏当前状态，风险统计与明细一致。

### SCN-TASK-001：项目经理分配任务，成员认领并推进工作

**Actors**：项目经理、团队长/小队长、开发人员、测试人员、外包员工。

**用户目标**：把需求或项目工作分配到唯一责任人，并通过任务状态反映实际执行情况。

**触发条件**：知脉同步需求类任务，或授权人员创建项目类/事务类任务。

**前置条件**：需求、项目、人员和组织关系有效。

**主流程**：

1. 需求类任务从知脉同步，知效保留源标识且不允许新建或删除源任务。
2. 项目经理分发任务，成员也可在规则允许时认领或流转负责人。
3. 开发任务和SIT测试任务分配后创建个人子任务，填写唯一责任人和工作量。
4. 子任务计划默认继承需求排期，授权人员可调整并留下变更记录。
5. 开发与对应SIT测试任务互相展示相关开发和测试人员。
6. 成员将个人子任务更新为进行中、阻塞、暂停或完成。
7. 团队按需要批量分发或批量更新任务，并查看逐项处理结果。

**Error Scenarios**：知效新建/删除知脉源任务的请求被拒绝；缺少责任人或工作量时不能完成分配；无权流转、非法状态变化和跨作用域批量更新被拒绝；批量操作不得用整体成功掩盖部分失败。

**Edge Cases**：事务类任务允许全员创建但永不计入工时；项目类任务可通过批准模板导入；负责人变更时历史责任人和操作记录必须保留。

**关联需求**：FR-TASK-001～008、FR-TASK-011、FR-TASK-014、FR-CROSS-003。

**成功结果**：每项执行工作有清晰责任人、状态和历史，任务类型对应的创建和计工规则得到执行。

### SCN-TASK-002：开发人员处理并确认安全要求

**Actors**：知安、开发人员、项目经理、安全管理人员。

**用户目标**：让安全要求随开发任务到达责任人，并将完成确认反馈给知安。

**触发条件**：知安完成威胁建模并推送安全场景、需求和设计要点。

**前置条件**：知安任务可映射到知效需求和开发任务。

**主流程**：

1. 系统按源任务标识幂等接收知安安全内容。
2. 涉及安全的开发任务显示醒目标识和只读安全内容。
3. 开发人员完成相应开发工作后显式确认。
4. 系统记录确认人和时间，并将确认结果幂等回传知安。
5. 项目经理和安全人员可查看同步与确认状态。

**Error Scenarios**：安全任务无法映射时进入同步异常；普通用户修改安全原文被拒绝；非责任开发人员确认时按权限拒绝；知安暂时不可用时保留待重试状态，不伪造成功。

**Edge Cases**：重复推送不产生重复安全任务；安全内容变更时保留版本；安全任务是否计入工时取决于`TBD-TASK-001`。

**关联需求**：FR-TASK-006、FR-TASK-008、FR-TASK-012～013、FR-CROSS-001～002。

**成功结果**：安全要求内容、开发确认和知安状态可以端到端追踪。

### SCN-WH-001：外包员工完成每日点亮与报工

**Actors**：外包员工、项目经理、考勤平台、调度作业J02/J03。

**用户目标**：基于真实考勤和实际进行中的任务生成默认报工，并在允许窗口内完成核对和调整。

**触发条件**：成员将个人子任务置为“进行中”；每日考勤同步与默认工时生成作业运行。

**前置条件**：外包人员、考勤和可报工开发任务映射有效。

**主流程**：

1. 个人子任务进入“进行中”时成为当日点亮任务，离开“进行中”时取消点亮。
2. 每人每天最多同时点亮5个可报工任务。
3. J02在03:00—06:00同步近14天（含）的最新考勤及补卡变化。
4. J03在考勤同步后为前一日生成默认报工。
5. 有N个点亮任务时，将考勤分钟分配为N条任务报工，并保持总分钟等于考勤分钟。
6. 人力外包当日没有点亮任务时，生成一条任务为空的异常报工。
7. 员工在列表或日历中查看本人报工，并可维护早于当天且位于14天窗口内的记录。
8. 每日总报工不得超过最新有效考勤，且只能关联已分配、归属系统一致、工作量未确认的开发任务。

**Error Scenarios**：第6个点亮任务被拒绝且原5个任务不变化；超考勤、当日、超过14天、已确认工作量或不符合任务资格的报工被拒绝；考勤人员无法映射时不生成无主报工；自动重算失败进入异常而不覆盖已锁定数据。

**Edge Cases**：补卡可能修改过去14天内考勤；无点亮任务的异常报工必须可检索；分钟不能整除任务数时的尾差顺序由`TBD-WH-001`确认；J03“08:00前完成”与J04“07:00写回”的顺序由`TBD-JOB-002`确认。

**关联需求**：FR-TASK-009～010、FR-WH-001～011。

**成功结果**：默认报工总分钟与考勤一致，所有调整可追踪，冻结和越界规则无法绕过。

### SCN-WH-002：管理者核查报工并同步知脉

**Actors**：行方负责人、中心经理、团队长、小队长、PMO、部落长、部门/行领导、知脉、J04/J05。

**用户目标**：在正确数据范围内核查外包报工，并在灰度期间避免知效和知微重复入账。

**触发条件**：管理者进入报工查看；J04执行每日写回；J05处理双系统数据。

**前置条件**：组织、负责人和外包人员关系有效；知脉工时接口可用。

**主流程**：

1. 系统按角色加载名下、本中心、本小队或批准的全局报工数据。
2. 管理者按人员、工号、公司、外包类型、方向、日期、需求和任务筛选。
3. 管理者查看报工变更历史和异常记录。
4. J04在每日07:00向知脉发送符合核算条件的报工，并记录逐条结果。
5. 失败记录按批准策略重试，耗尽后进入人工处理。
6. 灰度期间知脉先处理知效数据，再按工号+日期过滤知微重复数据。

**Error Scenarios**：越权查询被拒绝；知脉超时不把记录标为成功；部分批次失败必须返回成功/失败计数和明细；重复写回不产生重复业务结果。

**Edge Cases**：组织调整不能改变历史报工当时归属；跨系统同日多任务报工的去重粒度不得误删合法明细，具体契约在附件C确认。

**关联需求**：FR-WH-012～013、FR-CROSS-001～003、ROL-002。

**成功结果**：管理者只能看到获授权数据，知脉同步可追踪，双系统期间不重复核算。

### SCN-BASE-001：计算阶段时效并申请扣减

**Actors**：项目经理、部落长、PMO、管理层、J06。

**用户目标**：准确了解需求阶段停留，并对有业务依据的暂停时间发起审批扣减。

**触发条件**：J06每日运行、用户查询时效，或PM提交扣减申请。

**前置条件**：阶段进入/离开记录和业务日历有效。

**主流程**：

1. J06每日00:00计算在建需求各阶段停留天数。
2. 用户查询时，系统可按当前时点展示最新值，但不得重复累计。
3. PM选择阶段，填写扣减天数和原因并提交申请。
4. 部落长审批；只有批准后的扣减才影响有效停留天数。
5. 系统记录申请、审批、前后值和原因。
6. 需求回退并再次进入相同阶段时，系统累计各独立区间。

**Error Scenarios**：无原因、负数、超过累计停留或无权的扣减被拒绝；未批准申请不改变统计；同日重复计算不增加停留天数。

**Edge Cases**：周末、节假日和半天假的“业务天数”口径由`TBD-BASE-001`确认；扣减后有效天数不得小于0；多次进入同一阶段不得覆盖旧区间。

**关联需求**：FR-BASE-001～004。

**成功结果**：任意阶段的有效停留值都能由时间区间、业务日历和批准扣减重新计算得到。

### SCN-BASE-002：生成月度基线并识别加塞、取消和调出

**Actors**：PMO、项目经理、管理层、J07/J08/J09。

**用户目标**：形成稳定的月度交付考核快照，并识别5日之后进入或离开当月范围的需求。

**触发条件**：每月5日及6—30日的基线调度运行，或需求计划完成月份发生变化。

**前置条件**：需求计划完成日期和基线过滤配置有效。

**主流程**：

1. J07在每月5日00:00、12:00、16:00、22:00将符合条件的当月需求纳入基线。
2. J08在5日01:00、14:00、18:00、23:00取消计划完成月已不属于当月的基线。
3. J09在6—30日00:00、12:00、16:00、22:00识别未进基线但计划完成月为当月的加塞需求。
4. 5日后，已进基线但计划完成月变为非当月的需求标记为调出。
5. 首次进入基线或加塞且原计划为空时保存原计划，已有原计划不覆盖。
6. 取消时仅删除当月录入的原计划，跨月原计划保留。
7. 用户查看首次计划上线、后续计划变化和偏差分析。

**Error Scenarios**：重复作业不生成重复记录；无计划日期或配置不完整的需求进入待处理清单；人工修改历史快照被权限控制和审计阻止。

**Edge Cases**：5日和6日的日期边界严格按系统业务时区计算；跨月和跨年计划遵循目标月份；30日后至次月5日前的处理规则不得由实现自行推断，应按已批准调度规则执行。

**关联需求**：FR-BASE-005～011。

**成功结果**：每个需求在每个基线月的基线、取消、加塞和调出状态唯一且可还原。

### SCN-OPS-001：迁移知微存量数据并完成切换校验

**Actors**：迁移管理员、DBA、业务验收人员、测试、知微和数据仓库。

**用户目标**：在不破坏源数据和考核连续性的前提下，将范围内数据迁入知效并证明一致性。

**触发条件**：预迁移演练或正式迁移窗口开始。

**前置条件**：迁移范围、源快照、映射规则、冻结点和回滚方案已批准。

**主流程**：

1. 按组织/人员、需求/阶段、任务/协作、活跃报工、基线/考核顺序迁移。
2. 每批记录批次号、输入摘要、成功/失败数、检查点和操作者。
3. 中断后从已确认检查点续传，不重复创建对象。
4. 对记录数、关键字段、关联关系和业务汇总执行全量校验。
5. 至少抽样10%进行附加人工或独立复核。
6. 业务、数据、测试和运维共同签署对账证据。
7. 必要时按批次回滚，源数据保持不变。

**Error Scenarios**：异常记录未清零时禁止切换；关联完整性失败时回到修复/重跑流程；回滚请求无授权或缺少影响范围时拒绝执行。

**Edge Cases**：仅近14天活跃报工进入知效活动库，更早汇总进入批准的数据仓库；已归档需求进入归档存储；抽样通过不能替代全量校验。

**关联需求**：MIG-001～007、MIG-VAL-001～006。

**成功结果**：全部迁移范围都有明确结果，关键字段和汇总一致，迁移可重复、可续传、可回滚。

### SCN-OPS-002：按白名单灰度上线并在故障时回滚

**Actors**：发布经理、运维、PMO、外包员工、知效、知微、知脉。

**用户目标**：逐批切换报工用户，在严重故障时恢复知微承接并保全知效期间数据。

**触发条件**：批准的灰度批次开始，或达到回滚触发条件。

**前置条件**：白名单、监控、去重、补偿和回滚方案已演练。

**主流程**：

1. 发布人员按厂商、研发系统或部落配置白名单并记录变更。
2. 命中白名单的用户使用知效，未命中用户继续使用知微。
3. 知脉按已批准优先级接收知效和知微报工并去重。
4. 每批扩围前检查完整性、同步成功率、关键错误和用户反馈。
5. 达到严重故障门槛后停止扩围并作出回滚决策。
6. 回滚关闭知效入口，将报工流量恢复至知微。
7. 导出知效期间数据，经核对和审批后补偿，不自动覆盖未确认数据。

**Error Scenarios**：普通用户修改白名单被拒绝；回滚时旧系统不可用则进入应急升级而不是宣称成功；补偿数据存在冲突时暂停自动写入并人工处理。

**Edge Cases**：人员同时命中多个白名单规则时的优先级由`TBD-ROL-002`确认；白名单变更和用户提交并发时以请求受理时的生效版本为准；量化回滚门槛由`TBD-ROL-001`确认。

**关联需求**：ROL-001～006、NFR-OBS-001～002。

**成功结果**：灰度范围可证明、可监控、可撤销，回滚后业务可继续且期间数据不丢失。

## 6. Error Scenarios

## 9. Out-of-Scope Behavior

本期明确不要求系统提供以下行为：

- 在知效中重构知脉、知测、知安、知悉、考勤或数据仓库自身业务流程；
- 替代外部系统对需求主数据、测试结论、安全要求、身份、组织和考勤结果的主数据责任；
- 允许知效直接修改知脉来源需求/任务或知安安全要求原文；
- 支持公众互联网、匿名访问、独立用户名密码、离线客户端或原始需求未提出的移动端；
- 将自定义字段同步至外部系统；
- 允许用户在报工页面直接修改任务点亮状态；
- 迁移全部历史报工明细到知效活动库；仅近14天明细进入活动库，历史汇总按批准方案保存；
- 用PRD直接批准物理部署拓扑、采购规模、数据库分片键、密钥算法或生产凭据；
- 在未关闭TBD时由研发自行选择业务规则；
- 在没有环境、执行人、日期和证据编号的情况下把验收项标记为通过。
