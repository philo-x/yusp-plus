## 1. 数据模型目录

### 1.2 公共字段与复用边界

| 类别 | 规范 |
|---|---|
| 主键与关联 | 知效自建表主键及关联骨架对象的`user_id`、`org_id`、`pm_id`等统一使用`varchar(32)`；外部业务标识统一按`varchar(100)`起步并按源系统契约校验，不得强制截断。 |
| 租户与审计 | 自建表必须包含`data_tenant_id`及创建审计字段；可变对象包含修改审计字段，并发修改对象使用`version`乐观锁。业务唯一键和主要查询索引必须包含租户维度。 |
| 删除与历史 | 仅允许删除的主数据或业务数据使用`delete_flag`；事件、操作、阶段、迁移、基线变更和业务审计日志只追加，不提供逻辑或物理删除接口。 |
| 状态与时间 | 业务状态与删除标志分离；需比较先后的时间使用`datetime`，接口统一使用带时区ISO 8601，业务日期使用`YYYY-MM-DD`。 |
| JSON边界 | 原始载荷、前后值、动态配置可使用经GoldenDB验证的`json`；参与过滤、排序或统计的字段必须拆为有类型的列并建立适用索引。 |
| 骨架能力 | 字典、逻辑系统、访问日志、文件物理元数据和J01—J13调度定义优先复用基础骨架；知效自建关系、安全扫描结果、对象级重试/补偿和业务审计明细。 |
| 认证数据 | 统一认证使用知悉OAuth2；禁止启用本地密码登录或保存外部系统明文/可逆密码，迁移账号只允许使用不可登录的随机占位哈希。 |

| ID | 约束 | 验证方法 |
|---|---|---|

| DATA-RULE-003 | 项目、需求参与人、研发系统归属、自定义字段可见对象及个人筛选方案必须使用关系/配置表表达；全局字段最多20个、项目私有字段最多10个，以作用域配额槽位原子占用。 | 项目权限真值集、有效期、并发第10/11和20/21个字段测试 |

### 1.4 GoldenDB物理设计门禁

字段级逻辑模型、最低唯一键和查询索引以[表设计](../source/表设计.md)为输入，但关系说明不等同于批准物理外键。部署模式、分发键、全局主键/唯一索引、同域共置、广播小表、字符集/排序规则和JSON能力均须在目标GoldenDB 6.x实例评审验证；至少通过空库及重复建表、幂等消息、并发点亮、报工重算、基线调度、迁移断点续传和回滚演练后，方可批准物理DDL，见`TBD-DATA-002`。

## 2. 公共接口契约

### 2.1 计数和编号

本分册收录原接口分析中108个逻辑接口，编号`IF-001`至`IF-108`。编号是需求追踪编号，不等同于HTTP资源数量；除`IF-078`和`IF-094`按一个业务能力追踪一组动作外，每个编号对应一个Controller动作。实现和OpenAPI须分别定义每个动作，不得再以同一路径的`GET/POST/DELETE`组合表达一个逻辑接口。

适配基线以`.specify/memory/codebase-context.md`所核验的当前代码骨架为准：Controller以动作型POST为主（快照中254个`@PostMapping`、6个`@GetMapping`，无`@PutMapping`或`@DeleteMapping`），前端拦截器仅对POST业务体自动封装`{head, body}`。因此本目录优先保证与现有Controller、前端拦截器和异常响应链兼容，而不是引入一套平行的纯REST资源风格。

### 2.2 适用于所有接口的规则

| 项目 | 统一规则 |
|---|---|
| 传输 | 生产环境仅允许TLS；字符集UTF-8；时间使用带时区的ISO 8601格式，业务日期使用`YYYY-MM-DD`。 |
| 认证 | 除SSO发起/回调和受信系统回调外均要求统一认证；系统接口使用经安全批准的服务身份。 |
| 授权 | 查询按数据作用域裁剪；写操作同时校验功能权限、数据范围和对象当前状态。仅隐藏前端入口不构成授权。 |
| 租户 | 服务端从受信身份或会话解析租户并强制注入查询、唯一性校验和写入上下文，不接受客户端任意指定其他租户。 |
| HTTP与路径 | 业务JSON接口遵循现有Controller骨架，统一使用动作型`POST /api/<controller>/<action>`；查询使用`page/list/info/tree/stats`，写入使用`add/save/update/delete/confirm/sync`等动作名。不得新增`PUT`、`DELETE`或仅靠HTTP动词区分业务语义的同路径接口。SSO回调、OAuth2协议端点、文件上传下载及导入导出流是明确例外。 |
| 输入 | 业务JSON请求统一为`JClientReqEntity<T>`，报文为`{head, body}`；`head`沿用`reqJnls/pageId/transName/timestamp/nonce`等骨架字段，业务参数只放入`body`。查询模型使用`*Query`，命令模型使用现有`*Bo`、`*Dto`或`*Form`后缀；路径、分页、枚举、日期范围和业务字段均在Controller边界校验。 |
| 分页 | 分页查询的`*Query`继承或等价复用`PageQuery<T>`，使用`page`、`size`、`sort`，当前骨架默认第1页、每页10条；最大页大小和允许排序字段在`TBD-API-001`中固化。响应`body`使用MyBatis-Plus `Page<*Vo>`或经评审批准的等价分页投影。 |
| 输出 | 普通JSON响应统一为`JClientRspEntity<T>`；`head.retCode`、`head.retMsg`表达业务结果，`head.reqJnls`回传请求流水，`body`承载`*Vo`、`List<*Vo>`、`Page<*Vo>`或动作结果。不得再定义另一套`code/message/data`外壳。文件流和协议端点按各自媒体类型/协议返回。 |
| 错误 | 至少区分参数错误、未认证、无权限、对象不存在、状态冲突、重复请求、外部依赖失败和系统错误。普通业务及校验异常沿用`JClientRspEntity.buildFail`；依现有全局异常处理通常仍为HTTP 200并由`retCode`判定，OAuth2、认证失败、文件流等例外使用协议规定的HTTP状态。不得向客户端泄露堆栈或敏感配置。 |
| 幂等 | 业务接口以POST承载后，不得依赖HTTP方法的天然幂等语义；分页、列表、详情等查询动作必须只读，创建、同步、迁移、批量和调度触发动作必须校验业务幂等键，更新、删除和状态流转的重复调用应得到稳定结果。 |
| 并发 | 更新接口使用版本号或等价并发控制，冲突不得静默覆盖。 |
| 审计 | 特权查询、所有写操作、批量导入导出、同步配置、迁移、灰度和回滚均记录操作者、IP、对象、前后值/摘要、结果和追踪号；跨域记录写入`business_audit_log`。 |
| 重试 | 仅对明确可重试的超时/临时错误重试；次数与退避策略见`TBD-INT-002`，业务校验错误不得自动重试。 |
| 契约状态 | 下表为逻辑接口基线；精确请求/响应Schema、错误码和分页默认值由`TBD-API-001`关闭并固化为OpenAPI。 |
