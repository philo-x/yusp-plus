### 0.1 公共字段规范

除明确标记为“基础骨架复用表”的表外，本文所有知效自建表均应遵守以下约定。各表字段清单为突出业务字段，不再重复罗列所有公共字段；生成 DDL 时必须补齐。

| 类别 | 必备字段 | 规则 |
|------|----------|------|
| 租户隔离 | `data_tenant_id varchar(32)` | 非空，默认值与骨架一致；所有业务唯一键和主要查询索引应包含租户维度 |
| 创建审计 | `created_by varchar(32)`, `create_time datetime` | `created_by` 关联 `admin_sm_user.USER_ID`；系统任务使用约定的系统账号 |
| 修改审计 | `updated_by varchar(32)`, `update_time datetime` | 适用于可变业务表；事件、操作、变更日志表只追加，不更新 |
| 并发控制 | `version int` | 需求、任务、报工、基线、同步配置等并发修改对象采用乐观锁，初始值为 0 |
| 逻辑删除 | `delete_flag tinyint` | 仅用于允许删除的主数据/业务数据，`0` 正常、`1` 删除；审计和事件日志禁止逻辑删除接口 |

补充规则：

1. 业务状态字段与 `delete_flag` 分离，不得以同一个 `status` 同时表达业务状态和删除状态。
2. 时间字段统一按应用时区写入，接口层使用 ISO-8601；需要判断先后顺序的字段必须使用 `datetime`，不能只保存 `date`。
3. 新建表默认使用与目标实例统一的 `utf8mb4` 字符集和排序规则；与现有 `utf8_general_ci` 表关联时须通过兼容性测试，禁止依赖隐式字符集转换。
4. 关系说明不等同于直接创建物理外键。GoldenDB 分布式部署下，物理外键是否启用由 DBA 评审决定；未启用时必须有索引、应用校验和迁移完整性校验替代。
5. 原始载荷、前后值和动态配置可使用 GoldenDB 支持的 `json` 类型；需要过滤、排序或统计的业务字段不得只存 JSON。
6. `admin_sm_user.USER_PASSWORD` 为骨架非空字段，但知效仅使用知悉 OAuth2 统一认证。本地密码登录必须禁用，迁移用户使用不可登录的随机占位哈希，禁止保存外部系统明文或可逆密码。

### 0.2 基础能力复用边界

除下文逐项列出的 8 张身份组织核心表外，还复用以下骨架能力，但不计入知效自建表数量：

- `admin_sm_lookup_dict`：低频枚举和可调业务参数；复杂规则不得只存单一字典值。
- `admin_sm_logic_sys`：研发系统主数据；知效只扩展系统模块和人员归属关系。
- `admin_sm_log`：平台访问日志；不能代替要求保存前后值、原因、结果和追踪号的业务审计日志。
- `admin_file_upload_info`：文件物理元数据；需求附件表保存业务关系及安全扫描结果，不重复承担文件存储职责。
- `schedule_job`、`schedule_job_log`：J01—J13 调度定义与作业级运行日志；对象级失败、重试和补偿由同步事件表记录。

### 0.3 待业务确认项

1. 13 个标准阶段以规范化附件B表1为准，包含终态 `ARCHIVED`。`ARCHIVED` 是需求生命周期的终态阶段，不进入在建泳道；`requirement.archive_status` 仅作查询缓存，归档/取消归档仍须通过阶段实例和操作日志表达。
2. J03 要求 08:00 前生成默认报工，J04 暂定 07:00 写回知脉，存在先后冲突；调度时间不作为当前表结构的固定默认值。
3. `workload_confirm_status` 表示知脉“工作量已确认/封板”，不是知效本地报工审批。若后续引入本地审批，应单独增加审批单和审批历史，不复用该字段。
4. 默认工时均摊出现余数时，剩余分钟的分配顺序尚未确认；正式规则必须保证各报工记录分钟数之和等于考勤分钟数，且同一输入重跑结果一致。
5. 表数量按“8 张身份组织核心复用表 + 64 张知效自建表”统计；数据字典、日志、附件、逻辑系统和调度等其他复用表不计入。原 47/48/50/64 张总表数口径不再作为本稿基线。

### 4.8 GoldenDB 物理设计要求

1. **部署模式先决策**：目标实例若采用单分片，可沿用骨架的 MySQL/InnoDB 风格主键和索引；若采用多分片，必须在 DDL 评审中为每张表明确分发键、全局主键/全局唯一索引和同分片关联策略。
2. **同域数据共置**：需求、阶段、风险、参与人、计划变更、时效和基线优先按需求聚集；任务及其日志按任务聚集；报工、考勤和每日点亮优先支持“用户+日期”查询。若这些访问路径无法同时同分片，应通过全局索引或冗余快照减少跨分片 Join。
3. **小表策略**：字典、配置、字段定义、同步映射、业务日历等低频小表可作为广播/复制候选，最终以 DBA 批准语法为准。
4. **全局唯一性**：当分发键不包含业务唯一键时，`req_code`、`task_code`、来源对象ID和事件ID必须采用 GoldenDB 全局唯一能力或集中式发号/幂等服务保证，不能只依赖分片内唯一索引。
5. **JSON 使用边界**：JSON 可用于载荷快照、前后值和条件树；常用筛选字段必须拆列。上线前验证 JSON 函数、生成列/索引能力及 ORM 类型映射。
6. **字符集和大小写**：新表建议统一 `utf8mb4`；与骨架 `utf8_general_ci` 关联字段的字符集、长度和排序规则必须完全明确。Linux 环境禁止依赖表名大小写不敏感。
7. **DDL 验证门禁**：至少执行空库建表、重复建表、基础种子数据、幂等重复消息、2,000 TPS 并发点亮与报工上限竞争、补卡重算、基线四次调度、灰度多任务来源裁决、迁移断点续传/版本冲突/重复回滚、归档检索恢复演练，并保存目标 GoldenDB 版本、分发方案和验证结果。
