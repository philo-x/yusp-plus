# PERM-001 角色与数据权限管理 - 需求说明书 (requirements.md)

> **任务包编号**：PERM-001  
> **任务包名称**：角色与数据权限管理  
> **关联 PRD 章节**：第 3 章 (3.1 ~ 3.5), 2.1  

---

## 1. 功能需求 (Functional Requirements)

- **`FR-001`**：系统必须支持原始 PRD 定义的业务角色体系，包括：行领导、部门领导、部落长、中心经理、团队长、PMO、小队长、项目经理、系统设计师、开发人员、测试人员、外包员工、管理员及行方负责人。
- **`FR-002`**：系统必须支持需求管理视图下的管理层组织切换功能，管理层（行领导、部门领导、部落长、中心经理、团队长、PMO）默认加载本管辖主部落，支持自由切换至授权范围内的其他部落或下属小队，切换后顶部固定横幅与需求列表数据同步刷新。
- **`FR-003`**：系统必须支持需求管理视图下的执行层数据隔离，执行层角色（小队长、项目经理、开发人员、测试人员、外包员工）无组织切换能力，顶部横幅固定展示所属部落和小队的整体统计数据。
- **`FR-004`**：系统必须支持任务管理的多级数据隔离视图，支持部门领导/部落长查看本部门/部落全量待办；中心经理/团队长查看本部落/中心全量待办；PMO 查看所属部门/部落/中心全量待办；项目经理及开发/测试人员查看本人各类待办。
- **`FR-005`**：系统必须支持报工查看菜单的独立权限控制，该菜单仅向行方用户开放（行方负责人、中心经理/团队长、小队长、PMO/部落长/部门领导/行领导），外包人员不可见该查看菜单。
- **`FR-006`**：系统必须支持自定义字段的四级创建权限作用域控制，不同层级角色仅能创建对应作用域内的字段。

---

## 2. 非功能需求 (Non-Functional Requirements)

- **`NFR-001`**：权限校验切面与组织树获取接口响应时间不超过 100ms（95th Percentile）。
- **`NFR-002`**：用户组织角色与数据权限矩阵必须缓存于 Redis 中（`zx:user:perm:{user_id}`），降低 GoldenDB 数据库高频鉴权查询压力。
